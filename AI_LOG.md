## [2026-07-14] Milestone 0 smoke test — reasoning-model gotchas in llm.py
**Prompt/ask:** Run a live smoke test of `react_repro/llm.py::generate()` against z-ai/glm-5.2 via OpenRouter once the API key was filled in, to confirm the model responds and stop sequences are honored.
**Issue 1 — silent empty completions:** First call (`max_tokens=20`) returned `['']`. Raw API inspection showed `finish_reason: "length"` with `content: null` — z-ai/glm-5.2 is a reasoning model that spends its token budget on a hidden `reasoning` field before emitting `content`. At low max_tokens the reasoning phase alone exhausts the budget, so `content` never gets written.
**Fix 1:** Added `extra_body={"reasoning": {"enabled": False}}` to the OpenRouter request in `llm.py`. This suppresses the hidden CoT phase entirely — content now appears immediately (confirmed `finish_reason: stop`, `content: 'BANANA'`), and cost dropped ~19x (from $0.00044 to $0.000023 for the same prompt) since we don't pay for reasoning tokens we don't use. Also methodologically correct: ReAct/CoT already interleave explicit `Thought:` steps in the prompt, so a hidden reasoning phase on top is redundant and risks interfering with stop-sequence parsing of the `Action:`/`Observation:` structure.
**Issue 2 — n>1 silently ignored:** Requested `n=3` for CoT-SC-style majority-vote sampling; OpenRouter/provider (Decart) returned only 1 choice regardless of `n`, with no error — `len(response.choices) == 1` even when `n=3` was passed. This would have silently broken CoT-SC (majority vote over N samples degrades to a single sample, no error surfaced).
**Fix 2:** Rewrote `generate()` to never rely on server-side `n`. It now always requests `n=1` per call, and for `n>1` fans out `n` independent calls via a `ThreadPoolExecutor` and collects the individual results. Verified with `n=3` that 3 independent completions are returned (`count: 3`).
**Verification:** Re-ran three checks after both fixes — (1) basic short completion, (2) ReAct-style prompt with `\nObservation` stop sequence (truncation correct), (3) `n=3` sampling returning 3 independent completions. All passed.

**Issue 1 — mojibake in source JSON:** The official `prompts_naive.json` itself contains two encoding artifacts (an en-dash mis-encoded as literal bytes in "Arthur's Magazine (1844-\x80\x931846)" and "Pavel Urysohn ... â\x80\x93 August 17, 1924"). Confirmed this is present in the upstream source, not introduced by our transcription — left as-is per "verbatim" requirement; noted here so it isn't mistaken for a bug in our pipeline.
**Issue 2 — Wikipedia returned HTTP 403:** Initial `wiki_env.py` (built to mirror the official `wikienv.py` scraping approach: GET `en.wikipedia.org/w/index.php?search=...`, parse with BeautifulSoup) got blocked with a bare 403 and empty body — no working User-Agent header was set, and Wikipedia's bot-blocking has tightened since the paper's 2022 code was written.
**Fix 2:** Added a descriptive `User-Agent` header (`ReAct-Reproduction-Research/1.0 (educational assignment; contact: local)`) to the search request. Confirmed working: live search for "Colorado orogeny" now returns the exact sentence quoted in the paper's own exemplar ("The Colorado orogeny was an episode of mountain building...").
**Issue 3 — nav-chrome leaking into scraped page text:** On some pages (e.g. "High Plains" disambiguation), `soup.find_all("p") + soup.find_all("ul")` over the whole document also picked up Wikipedia's sidebar nav ("Main pageContentsCurrent events...", "Donate\n\nCreate account\n\nLog in") — this is markup drift since 2022, not something to faithfully replicate.
**Fix 3:** Scoped the search to `soup.find("div", {"id": "mw-content-text"})` before extracting `<p>`/`<ul>` tags, which contains only the article body. Verified all three paper exemplar scenarios post-fix: exact-match search, disambiguation ("may refer to:" redirect handling), and no-match-with-similar-suggestions fallback all match expected paper behavior.
**Assumption noted:** The official repo's `prompts_naive.json` ships `webact_simple6` (Act exemplars) but the notebook only defines an instruction preamble for ReAct, not Act. Constructed `ACT_INSTRUCTION` in `strategies.py` by removing the Thought-related line from the published ReAct instruction, since Act needs the same action-space description to parse correctly. Documented as an assumption, not verified against the original paper's exact Act prompt (not published).
