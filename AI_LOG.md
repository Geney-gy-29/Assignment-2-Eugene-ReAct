## [2026-07-14] Milestone 0 smoke test — reasoning-model gotchas in llm.py
**Prompt/ask:** Run a live smoke test of `react_repro/llm.py::generate()` against z-ai/glm-5.2 via OpenRouter once the API key was filled in, to confirm the model responds and stop sequences are honored.
**Issue 1 — silent empty completions:** First call (`max_tokens=20`) returned `['']`. Raw API inspection showed `finish_reason: "length"` with `content: null` — z-ai/glm-5.2 is a reasoning model that spends its token budget on a hidden `reasoning` field before emitting `content`. At low max_tokens the reasoning phase alone exhausts the budget, so `content` never gets written.
**Fix 1:** Added `extra_body={"reasoning": {"enabled": False}}` to the OpenRouter request in `llm.py`. This suppresses the hidden CoT phase entirely — content now appears immediately (confirmed `finish_reason: stop`, `content: 'BANANA'`), and cost dropped ~19x (from $0.00044 to $0.000023 for the same prompt) since we don't pay for reasoning tokens we don't use. Also methodologically correct: ReAct/CoT already interleave explicit `Thought:` steps in the prompt, so a hidden reasoning phase on top is redundant and risks interfering with stop-sequence parsing of the `Action:`/`Observation:` structure.
**Issue 2 — n>1 silently ignored:** Requested `n=3` for CoT-SC-style majority-vote sampling; OpenRouter/provider (Decart) returned only 1 choice regardless of `n`, with no error — `len(response.choices) == 1` even when `n=3` was passed. This would have silently broken CoT-SC (majority vote over N samples degrades to a single sample, no error surfaced).
**Fix 2:** Rewrote `generate()` to never rely on server-side `n`. It now always requests `n=1` per call, and for `n>1` fans out `n` independent calls via a `ThreadPoolExecutor` and collects the individual results. Verified with `n=3` that 3 independent completions are returned (`count: 3`).
**Verification:** Re-ran three checks after both fixes — (1) basic short completion, (2) ReAct-style prompt with `\nObservation` stop sequence (truncation correct), (3) `n=3` sampling returning 3 independent completions. All passed.

## [2026-07-14] Milestone 2 — FEVER prompts, domain-aware agent/strategies, n=10 gate
**Prompt/ask:** Build the FEVER pipeline reusing HotpotQA's WikiEnv/agent/strategies scaffolding: transcribe official FEVER few-shot exemplars, sample a dev set with the paper's exact methodology, make `agent.py`/`strategies.py`/`run.py` domain-aware (FEVER uses "Claim:" not "Question:", and its own baked-in instruction preamble), and run the n=10 gate across all 7 methods.
**Source decision:** Pulled `prompts/fever.json` directly from the official repo (github.com/ysymyth/ReAct), which ships `webthink_simple3` (ReAct), `cotqa_simple3` (CoT), `webact_simple3` (Act) — same naming convention as HotpotQA's `prompts_naive.json`. Extracted byte-exact into `react_repro/prompts/fever_{react,cot,act}_examples.txt`.
**Dataset:** Official `wrappers.py::FeverWrapper` loads `paper_dev.jsonl` (the FEVER shared-task's `paper_dev` split, 9,999 claims) — found and downloaded directly from `fever.ai/download/fever/paper_dev.jsonl` (the CMU/HF mirrors used for HotpotQA don't host FEVER). Applied the exact same paper methodology as HotpotQA (`random.Random(233).shuffle(idxs)` then take first 100) to `data/fever_dev_sample.jsonl` — confirmed from the official `FEVER.ipynb` cell, which uses `idxs = list(range(7405))` even though the actual file has 9,999 rows (upstream notebook bug/leftover from copy-pasting the HotpotQA cell); we shuffle over the true `len(rows)=9999` since reproducing the paper's actual FEVER accuracy numbers requires sampling from the real search space, not artificially truncating it to match a stale HotpotQA-shaped range.
**Issue — instruction/label mismatch across domains:** `agent.py::react()` and `strategies.py` (cot/cot_sc/act) were HotpotQA-hardcoded: prepended a HotpotQA-specific `INSTRUCTION` preamble and always labeled the query `"Question:"`. FEVER's official exemplars already bake in their own instruction line and use `"Claim:"`. Passing HotpotQA's `INSTRUCTION` on top of FEVER's baked-in instruction would have double-stated the task and mislabeled every claim as a "Question".
**Fix:** Added `instruction`/`query_label` parameters (default to HotpotQA's existing values, so HotpotQA callers are unaffected) threaded through `react()`, `react_strategy()`, `react_to_cotsc()`, `cotsc_to_react()`, `act()`, `cot()`, `cot_single()`, `cot_sc()`. `run.py::run_one()` now picks `query_label="Claim"` and `instruction=""` (both react and act) when `domain == "fever"`.
**Issue — `standard()` (no-reasoning baseline) returned free-text, not FEVER labels:** The HotpotQA-authored `standard()` prompt ("Answer the following question with a short answer") produced answers like "Sarah Michelle Gellar" for a claim that should have been judged SUPPORTS/REFUTES/NOT ENOUGH INFO — an EM-style prompt doesn't ask for a verification label, and `fever_acc()` would score every response as wrong regardless of whether the model's underlying judgment was correct.
**Fix:** Made `standard()` domain-aware — when `domain == "fever"`, prompts with a "Determine if the following Claim is SUPPORTS, REFUTES, or NOT ENOUGH INFO. Answer with exactly one of those three labels." instruction instead. Verified: predictions now come back as clean labels (`REFUTES`, `SUPPORTS`, etc.) matching `fever_acc()`'s expected format.
**Verification:** Smoke-tested standard/cot/act/react/react_cotsc/cotsc_react at n=2 each before the full gate — confirmed FEVER label parsing, live WikiEnv search/lookup reuse, and hybrid backoff logic all worked correctly on real FEVER claims.
**Gate result (n=10, all 7 methods, zero errors):** Standard 20%, CoT 40%, Act 40%, ReAct 30%, CoT-SC (n_sc=21) 50%, ReAct→CoT-SC 30% (backoff triggered 1/10), CoT-SC→ReAct 40% (backoff triggered 3/10). Both hybrid back-off mechanisms confirmed firing on real FEVER data (not dead code).
**Commit:** FEVER prompts (`fever_{react,cot,act}_examples.txt`), `data/fever_dev_sample.jsonl` (n=100, real FEVER `paper_dev` split, seed-233 shuffle prefix), and domain-aware `agent.py`/`strategies.py`/`run.py` changes; n=100 scale-up for both domains deferred to the analysis/report milestone per the time-boxed schedule.
